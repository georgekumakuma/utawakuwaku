# utawakuwaku レビュー（2025-12-13）

対象：Electron + localhost(Express) 配信のフロントで YouTube IFrame API を使い、動画の「区間再生」をプレイリスト化するアプリ。

- アプリ概要は [readme.md](readme.md) を参照
- Electron起動/配信は [main-electron.js](main-electron.js) を参照
- 区間再生/自動遷移の主因は [main.js](main.js) と [youtube.js](youtube.js)

---

## 結論（最重要）
既知不具合のうち、再現しやすく影響が大きい原因は以下の2つです。

1) **「停止したのに勝手に次へ進む」**：停止操作と `ENDED` イベント処理の競合で、遅延して届いたイベントが次曲遷移を起動できます。
- 自動次曲遷移が `event.data === 0` で無条件に走る： [main.js](main.js#L556-L576)
- 停止操作は `currentPlayingIdx = null` にするだけで、イベントを無視するガードがない： [main.js](main.js#L660-L671)

2) **「意図しないところで止まらない / 次の曲に送られる」**：区間監視の“手動シーク検出”が、バッファリング等の時間ジャンプも手動扱いしてしまい、区間終了が無効化されます。
- しきい値 `1.5s` 超の時間ジャンプで `endTimerDisabled = true`： [main.js](main.js#L360-L408)

加えて、**再生失敗時の扱いが弱い**（エラーコード取得・リトライ・理由表示がない）ため、「再生できずに次へ」の根本原因が見えづらくなっています。
- IFrame API の `onError` を購読していない： [youtube.js](youtube.js#L19-L44)

## 既知問題ごとの詳細（現状コードに基づく）

### A. 意図しないところで次の曲に送られる／停止しない

#### A-1. Stop後に `ENDED` で次曲へ行く（“停止しない”体感の原因）
- `onStateChange` で `ENDED(0)` を検知すると、**無条件で** `playNextSong()` を呼びます： [main.js](main.js#L563-L566)
- 一方で Stop ボタンは、動画停止と `currentPlayingIdx = null` をするだけです： [main.js](main.js#L660-L671)

**問題点**
- Stop直後またはStop近辺で、YouTube側から遅れて `ENDED` が届くと、意図せず次曲遷移が発火します。
- これは「停止しない」「勝手に次へ」の両方の体感につながります。

**改善方針**
- 自動遷移を許可するフラグ（例：`autoAdvanceEnabled`）と“再生セッションID”を導入し、Stopや手動操作後に **過去セッション由来のイベントを無視**する。

#### A-2. 区間終了判定（endTimer）の無効化が起きやすい
- 区間監視は `requestAnimationFrame` で `getCurrentTime()` を監視し、
  - `cur >= endSec - 0.3` でフェードアウト→次曲： [main.js](main.js#L387-L399)
  - ただし **時間ジャンプ**を検出すると「手動シーク」とみなし、`endTimerDisabled = true` にします： [main.js](main.js#L372-L385)

**問題点**
- バッファリング、ネットワーク揺れ、YouTube内部の再同期などでも `currentTime` が跳ねることがあり、ユーザーが触っていないのに「手動シーク」と誤判定 → 区間終了が無効化 → 曲が止まらずフル再生 → 末尾で `ENDED` → 次曲へ、という流れが起きえます。

**改善方針**
- “手動シーク検出”はプレイヤーの状態（`BUFFERING` 等）と組み合わせ、
  - バッファリング中の時間ジャンプは無視
  - 直近のユーザー操作（クリック/タッチ/キー/ドラッグ）をトリガにした場合のみ手動シークと判定
  - あるいは手動シーク検出そのものを廃止し、シーク後は一定時間だけ区間監視を再同期する

#### A-3. endSec の二重経路と責務の分散
- `playSongSection()` で `setVideo(... endSec: song.end ...)` を渡していますが、`youtube.js` 側は `endSec` を利用していません：
  - 呼び出し： [main.js](main.js#L270-L279)
  - 受け取り（未使用）： [youtube.js](youtube.js#L47-L67)

**問題点**
- 「区間終了はyoutube層でやるのか、main層でやるのか」が曖昧になり、バグ調査が難しくなります。

**改善方針**
- 区間終了の責務は1箇所に固定（推奨：`main.js` で統一）し、`endSec` は渡さない/または `youtube.js` で実装するなら `main.js` 側の監視を撤去する。

### B. 再生できずに次のリストに移ることがある
現状のプレイヤー実装は `onStateChange` のみ購読しており、`onError` がありません： [youtube.js](youtube.js#L19-L44)

**起きうるケース**
- 埋め込み不可・年齢/地域制限・権利ブロック等で `PLAYING` にならない
- 自動再生制約（特にモバイル）で `playVideo()` が拒否される

**現状の弱点**
- “失敗”の根拠（エラーコード/状態遷移ログ）が取れない
- そのため「なぜ次に行ったのか」「本当に失敗したのか」が判別しづらい

**改善方針**
- `youtube.js` に `onError` を追加し、上位へ通知（`setPlayerErrorCallback` 等）
- 失敗時は
  - 1回だけ再ロード（短いバックオフ）
  - だめなら理由付きでスキップ（UIに表示）

**確認/改善候補**

- `onError`（上記B）

## 付録：最小ログ設計（実装メモ）
ここは「修正を入れる前の調査」を加速するための、**最小限の観測ポイント**です。
実装は後回しでも、どこに何を出すかを先に決めておくと再現調査が一気に進みます。

### ログの基本方針
- **スイッチ式**：`localStorage.utawakuwaku_debug = "1"` のときだけ出す（普段は静か）
- **構造化**：1行1イベントで、JSON風にキーを揃える（後で比較しやすい）
- **理由（reason）を必ず付ける**：次曲遷移やタイマー無効化は「何が引き金か」を文字列で残す

推奨フォーマット例（console 1行）
```text
UWK {t: 173... , sess: 12, type: "requestNext", reason: "ended", idx: 3, videoId: "...", cur: 87.31, start: 75, end: 88, state: 0}
```

#### 有効化手順（Electron/Chrome）
1) DevTools Console で `localStorage.setItem('utawakuwaku_debug','1')`
2) リロード
3) Console の `UWK ...` 行を回収（必要なら `window.__uwkLogBuffer` も参照）


### ログを入れる場所（優先度順）
以下は「原因の切り分け」に直結します。

1) 次曲遷移
- `playNextSong()` を呼ぶ直前に必ずログ（reason付き）
  - 例：`reason = "ended" | "endTimer" | "userNext" | "startup"`

2) Stop / PlayPause / Prev / Next のクリック
- `btnStop` クリック：停止直後の `currentPlayingIdx` / `state` / `cur` をログ
- `btnPlayPause`：押下時の `state` と分岐（pause or play）をログ

3) YouTube `onStateChange`
- `event.data` をそのままログ（数値）
- 可能なら `cur`（getCurrentTime）も添える

4) endTimer（区間終了監視）
- `setEndTimer(endSec)` 呼び出し時にログ
- `endTimerDisabled=true` になった瞬間にログ（なぜ無効化したか、delta何秒か）
- `cur >= endSec - 0.3` でフェード開始した瞬間にログ（reason="endTimer")

5) YouTube `onError`（現状未実装）
- `onError` の code（数値）と videoId をログ
- 失敗時の次曲遷移にも必ず reason を付ける（reason="playerError")

### モバイルでのログ採取（重要）
スマホからの調査だと DevTools console が見づらいので、次のどちらかが有効です。
- 画面下に折りたたみ式の「ログパネル」を出す（直近N件だけ）
- `localStorage` にリングバッファ保存して、Exportボタンでテキスト化

## 付録：再現シナリオ（ログで白黒つける）

### シナリオ1：Stopしたのに次へ進む
目的：Stop後の遅延 `ENDED` が次曲遷移を起動しているか確認。

手順
1) 任意の曲を再生（区間あり）
2) すぐに Stop を押す（または終端付近で Stop）
3) 次曲に進んだら、その直前のログ順序を確認

期待するログのパターン（例）
- `btnStop` →（数百ms〜数秒後）`onStateChange state=0` → `requestNext reason="ended"`

結論
- これが出るなら、Stop後は `ENDED` を無視するガード（セッションID/フラグ）が必要。

### シナリオ2：意図せず区間停止が無効化される（手動シーク誤検知）
目的：バッファリング等の時間ジャンプが「手動シーク」と誤判定されているか確認。

手順（再現しやすい順）
1) ネットワークが不安定な環境で再生（Wi-Fi切替、テザリング等）
2) 再生中に画面を触らず待つ
3) 「手動シーク検出: 自動終了を無効化しました」が出たらログを見る

見るべきログ
- `endTimerDisabled` になった瞬間の `delta = |cur-lastCheckedTime|`
- 直前の `onStateChange` が `BUFFERING(3)` になっていないか

結論
- `BUFFERING` 直後の時間ジャンプで無効化しているなら、誤検知。

### シナリオ3：再生できずにスキップする
目的：埋め込み不可/制限/自動再生制約をエラーコードで切り分け。

手順
1) 再生できないことがある動画IDを使う（発生動画があればそれが最良）
2) `onError code` と `onStateChange` の流れを採取

結論
- `onError` が取れれば、スキップの正当性とUI表示（理由提示）が設計できる。
